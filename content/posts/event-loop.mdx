JS 是单线程的，但页面既能响应点击、又能跑定时器、还能等网络请求。这些"同时在跑"的东西，背后靠的就是事件循环。

**这篇笔记的用处：** 理解事件循环之前，遇到异步代码执行顺序不对、`setTimeout` 没按预期触发、Promise 和 setTimeout 谁先跑这类问题，只能靠猜。搞清楚之后能直接定位根因，也能更自信地选择 `setTimeout(fn, 0)`、`queueMicrotask`、还是 `Promise.resolve().then` 来控制执行时机。

**项目里最常碰到的场景：**
- 打字机动画、轮询状态——宏任务接力（`setTimeout` / `setInterval`）
- 数据请求、状态更新——微任务（`Promise.then` / `async/await`）
- 批量合并更新避免重复渲染——`queueMicrotask`
- 初始化时等 DOM 挂载完——`setTimeout(fn, 0)` 推到下一个宏任务

---

## 执行顺序

调用栈清空后，事件循环的调度规则：

```
主线程执行同步代码（直到调用栈空）
→ 清空微任务队列（全部跑完）
→ 从宏任务队列取一个任务
→ 再清空微任务队列
→ 再取宏任务
→ ……
```

## 宏任务和微任务

| 队列 | 对应代码 |
|------|---------|
| 宏任务 | `setTimeout`、`setInterval`、DOM 事件回调、网络请求回调 |
| 微任务 | `Promise.then`、`async/await`、`queueMicrotask`、`MutationObserver` |

```js
console.log('1')                                        // 同步
setTimeout(() => console.log('2'), 0)                   // 宏任务
Promise.resolve().then(() => console.log('3'))          // 微任务
console.log('4')                                        // 同步

// 输出：1 → 4 → 3 → 2
```

哪怕 `setTimeout(fn, 0)` 延迟是 0，它也是宏任务，要等微任务全清完才轮到它。

## async/await：也是微任务

`async/await` 本质是 `Promise` 的语法糖，`await` 之后的代码等价于 `.then()` 里的回调，进微任务队列：

```js
async function main() {
  console.log('A')        // 同步
  await Promise.resolve() // 把下面的代码放进微任务队列
  console.log('B')        // 微任务，等同步代码跑完再执行
}

console.log('start')
main()
console.log('end')

// 输出：start → A → end → B
```

## 项目里的例子：打字机效果

博客首页的打字机动画（[Typewriter.tsx](components/Hero/Typewriter.tsx)）就是事件循环驱动的：

```tsx
useEffect(() => {
  const target = roles[roleIdx]

  const timeout = setTimeout(() => {
    if (phase === 'typing') {
      if (display.length < target.length) {
        setDisplay(target.slice(0, display.length + 1))  // 打一个字
      } else {
        setTimeout(() => setPhase('deleting'), 2000)     // 停 2s 再删
      }
    } else {
      if (display.length > 0) {
        setDisplay(display.slice(0, -1))                 // 删一个字
      } else {
        setRoleIdx((i) => (i + 1) % roles.length)
        setPhase('typing')
      }
    }
  }, phase === 'typing' ? 80 : 40)

  return () => clearTimeout(timeout)
}, [display, phase, roleIdx])
```

每次只执行一个字符的更新，调用栈空下来时浏览器可以正常响应用户点击、滚动——这就是单线程不卡死的根本原因。每个 `setTimeout` 回调执行完，下一个才排队进来，形成"宏任务接力链"。

## 宏任务的常用场景

### setTimeout / setInterval — 定时执行

```js
// 延迟执行，不阻塞主线程
setTimeout(() => {
  console.log('1 秒后执行')
}, 1000)

// 轮询：每隔 3 秒检查一次状态
const timer = setInterval(async () => {
  const status = await checkOrderStatus(orderId)
  if (status === 'done') {
    clearInterval(timer)
    showSuccess()
  }
}, 3000)
```

`setTimeout(fn, 0)` 常用来把某段逻辑"推迟到当前同步代码执行完"：

```js
// 等 DOM 渲染完再操作
setTimeout(() => {
  document.getElementById('chart')?.scrollIntoView()
}, 0)
```

### DOM 事件回调 — 用户操作触发

```js
// 点击、输入、滚动都是宏任务
button.addEventListener('click', () => {
  console.log('点击了')
})

// 防抖：用 setTimeout 延迟处理，避免频繁触发
let debounceTimer
input.addEventListener('input', (e) => {
  clearTimeout(debounceTimer)
  debounceTimer = setTimeout(() => {
    search(e.target.value)  // 停止输入 300ms 后才真正搜索
  }, 300)
})
```

### MessageChannel — 比 setTimeout 更精确的异步

```js
// React 内部用 MessageChannel 实现任务调度
// 比 setTimeout(fn, 0) 延迟更短，适合高优先级的异步任务
const channel = new MessageChannel()
channel.port1.onmessage = () => {
  performWork()
}
channel.port2.postMessage(null)
```

## 微任务的常用场景

### Promise.then — 异步结果的后续处理

```js
fetch('/api/user')
  .then(res => res.json())       // 微任务 1
  .then(data => {                // 微任务 2
    renderUser(data)
  })
  .catch(err => {
    showError(err.message)
  })
```

### async/await — 更易读的微任务写法

```js
async function loadUser() {
  try {
    const res = await fetch('/api/user')
    const data = await res.json()
    renderUser(data)
  } catch (err) {
    showError(err.message)
  }
}
```

### queueMicrotask — 手动入队微任务

```js
// 批量更新，避免同步代码里多次触发
let pending = false
function scheduleUpdate() {
  if (pending) return
  pending = true
  queueMicrotask(() => {
    pending = false
    flushUpdates()  // 统一处理
  })
}
```

### MutationObserver — 监听 DOM 变化

```js
const observer = new MutationObserver((mutations) => {
  mutations.forEach(mutation => {
    console.log('DOM 变了：', mutation.type)
  })
})

observer.observe(document.getElementById('list'), {
  childList: true,
  subtree: true,
})

// 回调在当前宏任务结束后、下一个宏任务开始前执行（微任务时机）
```

## 总结

事件循环是 JS 单线程的调度核心：主线程空了，先把所有微任务跑完，再取一个宏任务，再清微任务，如此循环。理解这个顺序，就能解释"为什么 Promise 比 setTimeout 先执行"、"为什么 await 后面的代码不是立刻跑"，也能在需要精确控制执行时机时选对工具。
