## 技术选型

AccessToken 过期处理有几种思路：

- **跳登录页**：最简单，但用户体验差，操作中断
- **每个请求手动处理**：业务代码里到处写 `if (status === 401) refreshToken()`，冗余且容易遗漏
- **Axios 响应拦截器**：集中处理，业务代码完全无感知，是最合适的位置

选拦截器方案的理由很直接：**一处修改，全量生效**。所有经过同一个 axios 实例的请求都会自动获得续期能力，不需要改任何业务代码。

---

## 为了解决什么问题

JWT 认证体系里，AccessToken 通常有较短的有效期（15 分钟到 2 小时）。核心目标是：**用 RefreshToken 悄悄换一个新的 AccessToken，并把刚才失败的请求自动重发**，用户感知不到任何中断。

纯概念上不难。真正麻烦的是边界情况。

---

## 实际开发中碰到的问题

### 问题一：并发请求同时 401

页面加载时可能同时发出 5 个请求，如果这时候 Token 正好过期，5 个请求会同时收到 401。

最朴素的写法是每个 401 都去换一次 Token——这会触发 5 次 refresh 请求，通常 Token 服务会把后面几个视为非法并拒绝。

解法是 `isRefreshing` 标志位 + `failedQueue`：第一个 401 触发 refresh，后续的 401 全部排队等待新 Token：

```typescript
let isRefreshing = false
let failedQueue: Array<{
  resolve: (token: string) => void
  reject: (err: unknown) => void
}> = []

function processQueue(error: unknown, token: string | null) {
  failedQueue.forEach(({ resolve, reject }) => {
    if (error) reject(error)
    else resolve(token!)
  })
  failedQueue = []
}
```

### 问题二：重发请求触发拦截器死循环

第一个 401 换完 Token 后，需要把原始请求重发。但重发的请求会再次经过响应拦截器——如果它又收到 401（比如 Token 换了但接口服务还没刷新缓存），会再次触发 refresh，无限循环。

解法是在原始请求 config 上打标记 `_retry: true`，拦截器遇到已重试过的请求直接抛出：

```typescript
if (error.response?.status !== 401 || original._retry) {
  return Promise.reject(error)
}
original._retry = true
```

### 问题三：Native App 的 Token 不在 localStorage

H5 嵌入 Native App 时，Token 由客户端通过 JSBridge 注入，生命周期由 App 侧管理，`localStorage` 里根本没有 Token。调用 `localStorage.getItem('refresh_token')` 只会拿到 null。

另外，Native App 场景下"登录失败"的处理也不同——不能 `window.location.href = '/login'`，而是要调 `bridge.logout()` 触发 App 的登出流程。

适配方案是把 Token 操作抽成可替换的策略：

```typescript
// Web 环境
const tokenStrategy = {
  get: () => localStorage.getItem('access_token'),
  getRefresh: () => localStorage.getItem('refresh_token'),
  set: (token: string) => localStorage.setItem('access_token', token),
  onLogout: () => { window.location.href = '/login' },
}

// Native App 环境
const nativeTokenStrategy = {
  get: () => bridge.getTokenSync(),
  getRefresh: () => bridge.getRefreshTokenSync(),
  set: (token: string) => bridge.setToken(token),
  onLogout: () => bridge.logout(),
}
```

还加了**主动预刷新**：解析 JWT 的 `exp` 字段，在过期前 5 分钟主动换 Token，避免用户在操作关键流程时触发续期等待：

```typescript
const payload = JSON.parse(atob(token.split('.')[1]))
const msUntilExpiry = payload.exp * 1000 - Date.now()
setTimeout(() => proactiveRefresh(), msUntilExpiry - 5 * 60 * 1000)
```

## 完整方案代码

```typescript
// http.ts
let isRefreshing = false
let failedQueue: Array<{ resolve: (token: string) => void; reject: (err: unknown) => void }> = []

http.interceptors.response.use(
  res => res,
  async error => {
    const original = error.config

    if (error.response?.status !== 401 || original._retry) {
      return Promise.reject(error)
    }

    // refresh 接口本身 401，说明 refresh token 也过期了
    if (original.url === '/auth/refresh') {
      processQueue(error, null)
      tokenStrategy.onLogout()
      return Promise.reject(error)
    }

    original._retry = true

    if (isRefreshing) {
      return new Promise((resolve, reject) => {
        failedQueue.push({ resolve, reject })
      }).then(token => {
        original.headers.Authorization = `Bearer ${token}`
        return http(original)
      })
    }

    isRefreshing = true
    try {
      const { data } = await http.post('/auth/refresh', {
        refreshToken: tokenStrategy.getRefresh()
      })
      tokenStrategy.set(data.accessToken)
      processQueue(null, data.accessToken)
      original.headers.Authorization = `Bearer ${data.accessToken}`
      return http(original)
    } catch (err) {
      processQueue(err, null)
      tokenStrategy.onLogout()
      return Promise.reject(err)
    } finally {
      isRefreshing = false
    }
  }
)
```

主流程其实并不复杂，三个边界情况（并发、死循环、Native 适配）才是需要仔细处理的地方。
