## 核心差异在哪

用 React 做 SPA 和用 Next.js，底层都是 React，但思路差很多：

| | React SPA | Next.js |
|---|---|---|
| 路由 | react-router，手动配置 | 文件即路由，自动生成 |
| 首屏 | 先下载 JS，再渲染 | 服务端直出 HTML |
| 数据获取 | `useEffect` 里 fetch | `async` 服务端组件 / `getServerSideProps` |
| 部署 | 静态文件就行 | 需要 Node 服务（或 Vercel） |

---

## 路由：文件系统就是路由表

React SPA 里你要手写：
```jsx
<Route path="/blog/:slug" element={<BlogPost />} />
```

Next.js App Router 里你只需要建文件：
```
app/
  blog/
    [slug]/
      page.tsx   ← 对应 /blog/任意内容
```

`[slug]` 就是动态参数，组件里直接用：
```tsx
export default function BlogPost({ params }: { params: { slug: string } }) {
  return <div>{params.slug}</div>
}
```

## 服务端组件 vs 客户端组件

这是 Next.js App Router 最核心的概念，也最容易混。

**服务端组件（默认）：** 在服务器上跑，可以直接 `await` 数据，不能用 `useState` / `useEffect` / 事件监听。

```tsx
// 这是服务端组件，直接 async await
export default async function Page() {
  const data = await fetch('https://api.example.com/data').then(r => r.json())
  return <div>{data.title}</div>
}
```

**客户端组件：** 文件顶部加 `'use client'`，才能用 hooks 和事件。

```tsx
'use client'
import { useState } from 'react'

export default function Counter() {
  const [n, setN] = useState(0)
  return <button onClick={() => setN(n + 1)}>{n}</button>
}
```

**我的理解：** 把页面想成两层。数据层（fetch、数据库查询）放服务端组件；交互层（按钮、动画、表单）放客户端组件。

## 静态导出（这个博客就是这么部署的）

不需要 Node 服务的话，可以纯静态导出：

```javascript
// next.config.js
module.exports = {
  output: 'export',  // 生成静态文件
  trailingSlash: true,
}
```

跑 `next build` 会生成 `out/` 目录，里面是完整的 HTML + JS + CSS，扔到任何静态托管（GitHub Pages / 腾讯云 COS / Vercel）都能用。

限制是：不能用需要运行时服务端的功能（`getServerSideProps`、API Routes 等）。

## 几个刚开始容易搞错的地方

**1. Link 组件代替 a 标签**
```tsx
// ❌ 这样会全页刷新
<a href="/blog">博客</a>

// ✅ 客户端跳转
import Link from 'next/link'
<Link href="/blog">博客</Link>
```

**2. Image 组件在静态导出要关优化**
```javascript
images: { unoptimized: true }
```
静态导出没有服务器，图片优化（格式转换、按需裁剪）跑不了，要关掉，不然 build 会报错。

**3. basePath 的坑**
如果部署在子路径（比如 `/blog/`），`basePath` 设了之后 `Link` 和 `next/image` 会自动加前缀，但 `<img src="/xxx">` 不会——要手动拼 `${process.env.NEXT_PUBLIC_BASE_PATH}/xxx`。

---

总体感觉 Next.js 上手不难，习惯服务端/客户端组件的区分之后思路就顺了。它强制你思考"这块逻辑跑在哪"，反而比 SPA 的思路更清晰。
