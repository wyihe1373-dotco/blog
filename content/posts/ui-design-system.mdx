## 技术选型

业务组件库的选型通常在两个方向之间：

- **直接二次封装 Ant Design / MUI**：快，但 UI 风格受约束，移动端 H5 场景下这些库的体积和交互都不太合适
- **从零自建**：灵活，但成本高，容易烂尾

我们选择了中间路线：**基础逻辑自建，样式体系基于 CSS 变量**。不依赖具体 UI 库，但也不重复造基础能力（比如 `useControllableValue` 用 ahooks 的，动画用 Framer Motion 的）。

自建的核心理由是 H5 场景的特殊性：组件需要适配 Native 手势、底部安全区、WebView 的各种渲染怪癖，直接套 PC 端组件库适配成本反而更高。

## 为了解决什么问题

项目有 36+ 个 H5 app，早期每个 app 各自实现"筛选标签"这类交互：

```tsx
// app A 里
<div className="chip-list">
  {options.map(o => (
    <span
      className={selected === o.value ? 'chip chip--active' : 'chip'}
      onClick={() => setSelected(o.value)}
    >
      {o.label}
    </span>
  ))}
</div>

// app B 里（另一个人写的）
<div className="filter-tags">
  {items.map(item => (
    <button
      style={{ background: activeId === item.id ? '#6366f1' : '#1e1e2e' }}
      onClick={() => setActiveId(item.id)}
    >
      {item.name}
    </button>
  ))}
</div>
```

同一个交互，两套实现，样式不一致，逻辑无法复用——而且这些代码里**没有一行在解决业务问题**，全是 UI 状态管理的噪音。

## 实际开发中碰到的问题

### 问题一：受控模式写太死，调用方痛苦

第一版 FilterChips 只支持受控模式：

```tsx
// 只能这样用，必须外部管 state
const [selected, setSelected] = useState('all')
<FilterChips value={selected} onChange={setSelected} options={options} />
```

调用方反馈：很多场景根本不需要外部感知选中状态（比如纯筛选 UI，结果通过 `onSubmit` 一次性拿），每次都要声明 `useState` 很繁琐。

解法是同时支持受控和非受控，用 `useControllableValue`（ahooks）抹平两种模式：

```tsx
const [activeKey, onChange] = useControllableValue<string>(props, {
  defaultValuePropName: 'defaultValue',
  valuePropName: 'value',
  trigger: 'onChange',
})
// 传了 value → 受控；只传 defaultValue → 非受控；都不传 → 内部管理
```

### 问题二：弹窗关闭动画消失太快

第一版 Dialog 用了最简单的条件渲染：

```tsx
if (!open) return null
```

关闭时 DOM 直接消失，关闭动画根本来不及播放。

问题在于：关闭动画需要 DOM 存在，但动画结束后才能卸载 DOM。这是两个不同时机，一个 `open` prop 控制不了两件事。

解法是用两个状态分管：`mounted` 控制 DOM 是否存在，`isClosing` 触发关闭动画的 CSS class：

```tsx
const [mounted, setMounted] = useState(false)
const [isClosing, setIsClosing] = useState(false)

useEffect(() => {
  if (open) {
    setMounted(true)
    setIsClosing(false)
    document.body.style.overflow = 'hidden'  // 锁定背景滚动
  } else if (mounted) {
    setIsClosing(true)
    const timer = setTimeout(() => {
      setMounted(false)
      document.body.style.overflow = ''
      afterClose?.()            // 通知调用方"动画结束了"
    }, 300)                     // 与 CSS transition 时长保持一致
    return () => clearTimeout(timer)
  }
}, [open])

useEffect(() => () => { document.body.style.overflow = '' }, [])  // 组件卸载兜底
```

关闭流程变成：`open=false` → `isClosing=true`（触发动画）→ 300ms 后 `mounted=false`（卸载 DOM）。

### 问题三：文字长短不一，固定字号会溢出

筛选标签的文字长度差异很大（"今日" vs "近三个月"），固定字号在短选项上正常，在长选项上溢出或截断。

用 Canvas 的 `measureText` 在渲染前测量，找到合适的字号：

```tsx
useLayoutEffect(() => {
  if (!adaptive || typeof title !== 'string') return
  const el = containerRef.current
  if (!el || el.clientWidth === 0) return  // 容器未布局，跳过

  const canvas = document.createElement('canvas')
  const ctx = canvas.getContext('2d')!
  const { fontFamily, fontWeight } = window.getComputedStyle(el)

  const fits = (size: number) => {
    ctx.font = `${fontWeight} ${size}px ${fontFamily}`
    return ctx.measureText(title).width <= el.clientWidth
  }
  setFontSize(fits(16) ? 16 : fits(14) ? 14 : 12)
}, [title, adaptive])
```

用 `useLayoutEffect` 而非 `useEffect`，确保在浏览器绘制**前**完成字号调整，避免先渲染大字再闪缩小的抖动。

## 几条通用设计原则

回到这些具体问题，背后有几个可以复用的判断框架：

**受控与非受控都要支持**。调用方有时需要外部控制状态，有时不需要。强制受控会增加无谓的样板代码。

**时序问题要用多个状态管理**。一个 prop 控制不了两个不同时机的事情（比如"开始关闭"和"卸载 DOM"）。

**`data-*` 属性比动态 className 更可维护**。组件内部的状态变化用 `data-active`、`data-disabled` 驱动样式，CSS 里用属性选择器，比拼接 className 字符串更清晰也更容易扩展。

**接口比实现更难改**。Props 类型一旦发布、调用方多了就很难收回。先想清楚接口（参数名、受控模式、事件回调），再写实现。
