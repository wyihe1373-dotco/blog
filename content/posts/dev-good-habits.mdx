代码写出来不只是给机器执行的，更多时候是给人读的——包括三个月后的自己。这篇记录我在日常开发中养成的几个习惯，重点是落地性，不讲大道理。

## 一、注释：让编辑器帮你记住参数

### JSDoc / TSDoc

函数加上 JSDoc，鼠标悬停就能看到参数说明，不用跳转到定义处翻代码。

```ts
/**
 * 格式化金额，保留两位小数，带千分位
 * @param value 原始金额（分或元，由 unit 控制）
 * @param unit  'fen' 表示输入为分，自动除以 100；默认 'yuan'
 * @returns 格式化后的字符串，如 "1,234.56"
 */
export function formatAmount(value: number, unit: 'fen' | 'yuan' = 'yuan'): string {
  const yuan = unit === 'fen' ? value / 100 : value
  return yuan.toLocaleString('zh-CN', { minimumFractionDigits: 2, maximumFractionDigits: 2 })
}
```

调用处悬停 `formatAmount`，VS Code / WebStorm 会弹出完整说明，省去反复查函数签名的时间。

### 关键决策加注释

逻辑清晰的代码不需要注释，但**非显而易见的决策**一定要写：

```ts
// 后端返回的 status 用数字，0 = 待审核，1 = 通过，2 = 拒绝
// 注意：这里故意不用枚举，因为后端有时会加新状态，枚举会漏 case
const STATUS_LABEL: Record<number, string> = {
  0: '待审核',
  1: '已通过',
  2: '已拒绝',
}

// Safari 15 不支持 structuredClone，降级用 JSON 序列化
const deepClone = typeof structuredClone === 'function'
  ? structuredClone
  : <T>(obj: T): T => JSON.parse(JSON.stringify(obj))
```

## 二、消灭魔法数字

代码里裸写数字、字符串常量，三个月后自己都不知道它代表什么。

**问题代码：**

```ts
if (user.role === 2) { ... }
setTimeout(refresh, 300000)
el.style.zIndex = '9999'
```

**改进：**

```ts
// constants/user.ts
export const USER_ROLE = {
  VISITOR: 0,
  MEMBER: 1,
  ADMIN: 2,
} as const

// constants/time.ts
export const MS = {
  SECOND: 1_000,
  MINUTE: 60_000,
  HOUR: 3_600_000,
} as const

// constants/zIndex.ts
export const Z_INDEX = {
  MODAL: 1000,
  TOAST: 1100,
  TOOLTIP: 1200,
} as const
```

```ts
if (user.role === USER_ROLE.ADMIN) { ... }
setTimeout(refresh, 5 * MS.MINUTE)
el.style.zIndex = String(Z_INDEX.MODAL)
```

常量改名或调整时，一处修改全局生效，比全局搜索替换 `9999` 安全得多。

## 三、高频业务方法封装

每个项目都有一些散落各处、反复手写的逻辑。把它们收进 `utils/` 或 `composables/`，后期维护时改一处即全改。

### 典型场景：请求状态

```ts
// composables/useRequest.ts
import { ref } from 'vue'

export function useRequest<T>(fn: () => Promise<T>) {
  const data = ref<T | null>(null)
  const loading = ref(false)
  const error = ref<Error | null>(null)

  async function run() {
    loading.value = true
    error.value = null
    try {
      data.value = await fn()
    } catch (e) {
      error.value = e as Error
    } finally {
      loading.value = false
    }
  }

  return { data, loading, error, run }
}
```

业务侧：

```ts
const { data: userInfo, loading, run: fetchUser } = useRequest(() => api.getUser(id))
```

不再每个组件各自维护 `loading / error` 状态，也不再忘记 `finally` 重置 loading。

### 典型场景：防抖搜索

```ts
// utils/debounce.ts
export function debounce<T extends (...args: unknown[]) => void>(fn: T, delay: number): T {
  let timer: ReturnType<typeof setTimeout>
  return ((...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }) as T
}
```

```ts
const handleSearch = debounce((keyword: string) => fetchResults(keyword), 300)
```

## 四、Commit 信息写给人看

一个月后 `git log` 能快速定位改了什么，靠的是写得清楚的 commit 信息。

```bash
# 好的 commit
feat: 订单列表新增按状态筛选功能
fix: 修复 Safari 下日期选择器无法关闭的问题
refactor: 将金额格式化逻辑抽离为 formatAmount 工具函数
docs: 补充接口鉴权方式说明

# 差的 commit
fix bug
update
aaa
WIP
```

类型前缀约定：

| 类型 | 含义 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改功能） |
| `perf` | 性能优化 |
| `docs` | 文档 |
| `chore` | 构建 / 依赖 / 配置 |

## 五、技术文档：决策留痕

代码可以解释「怎么做」，但解释不了「为什么这么做」。技术文档的价值在于记录**当时的上下文和权衡**。

值得记录的内容：

- 为什么选了 A 方案而不是 B？（比如选了 `react-query` 而不是自己封装请求）
- 有哪些方案被否定了，原因是什么？
- 某个 API 的奇怪用法，背后是什么限制？
- 第三方依赖有哪些坑？

格式不重要，能找到、能看懂就行。可以是 `docs/` 目录下的 markdown，可以是 Notion，可以是 README 里的一段话。

```markdown
## 为什么用 sips 而不是 sharp 转换图片？

项目部署在 GitHub Actions (ubuntu)，sharp 在 macOS 交叉编译时偶发 native binding 错误。
调查后发现 GitHub Actions 的 ubuntu runner 已预装 ImageMagick，改用 `convert` 命令更稳定。
本地 macOS 开发仍用 sips（内置，无需安装）。
```

## 六、问题记录：别让 Bug 白踩

遇到难搞的问题，解决后记录下来：症状 → 根因 → 解法。

```markdown
## Safari 15 `position: sticky` 失效

**症状：** 表格头部在 Safari 15 下滚动时不吸顶，Chrome / Firefox 正常。

**根因：** 父容器设置了 `overflow: auto`，sticky 在 overflow 非 visible 的祖先容器内失效。

**解法：** 移除父容器的 `overflow: auto`，改用 JS 监听 scroll 事件手动吸顶；
或将 sticky 元素提到 overflow 容器之外。

**参考：** https://bugs.webkit.org/show_bug.cgi?id=xxxxxxx
```

这类记录的最大价值不是给自己看，是**队友下次遇到同样的坑可以直接搜到答案**，不用再花两小时排查。

## 小结

这些习惯没有一个有即时收益，但每一个都在降低未来的维护成本：

| 习惯 | 解决的问题 |
|------|------------|
| JSDoc 注释 | 不用跳转查函数签名 |
| 消灭魔法数字 | 改一处全局生效，不漏改 |
| 工具函数封装 | 逻辑改一处，不是改十处 |
| 规范 commit | git log 能当变更日志用 |
| 技术文档 | 决策有据可查，新人上手快 |
| 问题记录 | 同类 Bug 不重复踩 |

维护性好的代码，核心是「对未来的自己和队友友好」。
