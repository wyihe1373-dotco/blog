## 为什么用 Monorepo

项目有 36+ 个 H5 app，早期每个 app 都是独立仓库。共享代码走 npm 包，改动要先发版，消费方再升级，联调一个改动要来回跑好几次。版本碎片化严重——10 个 app 可能依赖同一个包的 10 个不同版本。

Monorepo 的核心优势是：共享包直接用 `workspace:*` 引用，改动即时生效，不需要发包流程。

## 从零搭建

### 第一步：配置 Yarn Workspaces

根目录 `package.json` 声明所有工作区：

```json
{
  "name": "h5hub",
  "private": true,
  "workspaces": [
    "packages/*",
    "apps/*",
    "sdk/*"
  ]
}
```

`yarn install` 会把所有包的依赖提升到根目录 `node_modules`，各包之间通过软链接互相引用。

### 第二步：TypeScript 路径映射

根目录 `tsconfig.json` 配置路径别名，让 `@pkgs/xxx` 指向 `packages/xxx`：

```json
{
  "compilerOptions": {
    "paths": {
      "@pkgs/*": ["packages/*"]
    },
    "baseUrl": "."
  }
}
```

这样 `import { postV1FundList } from '@pkgs/client'` 在 TypeScript 层面会正确解析到 `packages/client/src/index.ts`，有完整的类型提示。

Vite 侧用 `vite-tsconfig-paths` 插件读取这个映射，构建时自动处理模块解析：

```typescript
// vite.config.ts
import tsconfigPaths from 'vite-tsconfig-paths'

export default defineConfig({
  plugins: [tsconfigPaths()]
})
```

### 目录结构

```
h5hub/
├── apps/               # 36+ H5 应用
│   ├── fund/           # 基金 app
│   ├── cert/           # 证书
│   ├── auth/           # 登录
│   └── ...
├── packages/           # 共享包（约 39 个）
│   ├── client/         # API 调用函数（codegen 生成）
│   ├── components/     # 公共 UI 组件
│   ├── devkit/         # 构建工具包
│   ├── constants/      # 业务常量
│   ├── i18n/           # 国际化
│   └── ...
└── sdk/                # 底层 SDK
```

## 共享包的两种模式

### 源码直引（不需要构建）

`@pkgs/client`、`@pkgs/components` 等大部分包直接把 `main` 指向源码：

```json
{
  "name": "@pkgs/components",
  "main": "index.ts",
  "types": "index.ts"
}
```

应用通过 `@pkgs/components` 引入时，Vite 找到 `packages/components/index.ts`，和引入普通文件没有区别，由 Vite 一起编译。

好处是改了包里的代码，应用热更新立刻生效，不需要预先跑 `build`。

### 需要预先构建的包

`@pkgs/devkit` 是个例外——它的输出是 Vite 插件，被 `vite.config.ts` 在启动阶段引入：

```json
{
  "name": "@pkgs/devkit",
  "main": "./output/index.js",
  "types": "./output/index.d.ts"
}
```

`vite.config.ts` 运行在 Node.js 环境，Vite 还没启动、没法做模块转换，所以 devkit 必须提前编译成 CommonJS。开发前需要先跑一次：

```bash
yarn build devkit  # 编译 devkit
yarn dev fund      # 再启动应用
```

## 公共 Vite 插件：@pkgs/devkit

每个 app 的 `vite.config.ts` 其实只有五行：

```typescript
import { defineConfig } from 'vite'
import { vitePluginH5Hub } from '@pkgs/devkit'

export default defineConfig(() => ({
  plugins: [vitePluginH5Hub()]
}))
```

所有的公共配置都封装在 `vitePluginH5Hub()` 里，包括：

**React + 路径解析：**

```typescript
return [
  ...react(),              // React Fast Refresh
  tsconfigPaths(),         // @pkgs/* 路径映射
  legacyConfigPlugin,      // 兼容 Chrome 60+
  commonPluginConfig,      // 代理 + 主题注入
]
```

**多环境代理配置：**根据 `proxy` 环境变量自动切换请求目标：

```typescript
const getProxyTarget = (proxy: string) => {
  switch (proxy) {
    case 'sit':    return 'https://m.sit.example.com'
    case 'canary': return 'https://m.canary.example.com'
    default:       return 'https://m.example.com'
  }
}
```

跑 `proxy=sit yarn dev fund` 就能把所有 `/api` 请求指向 SIT 环境，不需要每个 app 各自配。

**主题注入：**`transformIndexHtml` 钩子在 HTML 的 `</head>` 前注入主题检测脚本，在 App 内打开时自动读取 UA 里的主题标识设置主题：

```typescript
async transformIndexHtml(html) {
  const scriptTag = `<script>
    const mode = getThemeModeByQuery() || getThemeModeByUA() || 'light'
    document.documentElement.setAttribute('theme', mode)
    document.documentElement.classList.add('theme-' + mode)
  </script>`
  return html.replace('</head>', `${scriptTag}</head>`)
}
```

没有 devkit 之前，每个 app 都要维护这一百多行配置，而且难以同步更新。

## 公共 UI 组件：@pkgs/components

`@pkgs/components` 封装了项目里高频使用的 UI 组件，基于 antd-mobile 二次封装，加了项目特有的主题和交互逻辑：

```
packages/components/
├── src/
│   ├── popup/       # 弹出层（内置 Teleport to body）
│   ├── picker/      # 选择器
│   ├── date-picker/ # 日期选择器
│   ├── loading/     # 全局 Loading
│   └── layout/      # 页面布局
├── index.ts         # 统一导出
└── package.json
```

应用里直接 import：

```typescript
import { Popup, Loading } from '@pkgs/components'
```

修改 `@pkgs/components` 里的组件，所有引用它的 app 热更新立刻反映，不需要任何发包流程。

## 日常工作流

`yarn dev fund` 的背后：

```bash
# 等价于直接跑 vite，但 CLI 先检查前置依赖
yarn dev fund

# CLI 内部逻辑：
# 1. 读取 apps/fund/package.json 的 buildDependencies 字段
# 2. 检查 packages/devkit/output/ 是否存在
# 3. 如果缺失，先自动 build devkit
# 4. 再启动 fund 的 dev server
```

`buildDependencies` 字段声明构建前置依赖，而不是运行时依赖：

```json
{
  "name": "@apps/fund",
  "buildDependencies": ["devkit"]
}
```

新人 clone 仓库后第一次跑 `yarn dev fund`，不需要看文档，CLI 自动处理依赖顺序。

## 三个绕不开的工程化问题

### 构建顺序

Yarn Workspaces 的 `yarn workspaces run build` 是并行的，不保证顺序。`@pkgs/devkit` 还没编译好，`@apps/fund` 就已经开始找 `output/index.js`——必然报错。

用 `buildDependencies` + 自定义 CLI 解决：

```typescript
const needBuild = buildDependencies?.filter(dep => !checkBuildDependencie(dep)) ?? []
if (needBuild.length) {
  // 按顺序先 build 前置依赖
  const cmds = needBuild.map((d, i) =>
    [i === 0 ? null : 'yarn', 'build', d].filter(Boolean)
  )
  cmds.push(['yarn', 'workspace', app, 'run', action])
  yarnRun(cmds.flat(), envVars)
}
```

### 架构边界

SDK 层是基础设施，不应该引用业务 package，否则会形成循环依赖。这个约束只写在文档里不够，用 `import-lint` 把规则写成配置，提交时自动扫描暂存文件：

```json
{
  "rules": {
    "sdk/app": {
      "mode": "blacklist",
      "disallow": ["@pkgs/*"],
      "allow": ["@pkgs/constants", "@pkgs/theme"]
    }
  }
}
```

### 大仓 lint 速度

Prettier 和 ESLint 是 Node.js 实现的，单文件处理快，但大仓 100+ 文件全量扫描需要十几秒。pre-commit hook 每次提交都触发，体验很差，开发时容易选择用 `--no-verify` 绕过，规范形同虚设。

Oxc 是用 Rust 写的同类工具——`oxfmt` 对应 Prettier 做格式化，`oxlint` 对应 ESLint 做代码检查。Rust 的执行效率比 Node.js 高得多，同样的文件集速度提升 10–50 倍。pre-commit 从十几秒缩短到不到一秒。

迁移成本主要是规则兼容性——oxlint 不支持所有 ESLint 插件，迁移前需要核对规则覆盖范围。格式化规则基本兼容，oxfmt 的输出和 Prettier 差异很小。

---

核心思路：把人工约束变成工具约束。文档里写的规范，靠机器来执行。
