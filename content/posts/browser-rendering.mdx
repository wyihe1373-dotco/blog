## 为什么值得理解这个过程

这个流程的实际价值是：**遇到问题时知道从哪里排查，平时写代码时知道该注意什么**。

页面白屏？可能是 DNS 解析慢、TCP 连接慢、服务器响应慢、JS 执行阻塞渲染——不理解流程就只能盲猜。

下面按照实际发生的顺序过一遍，每一步说清楚做了什么、可能出什么问题。

---

## 1. URL 解析

浏览器先解析输入的 URL，拆出协议、域名、路径、参数：

```
https://example.com/path?key=value#anchor
  协议     域名       路径   查询参数   锚点
```

如果输入的不是合法 URL（比如少了协议），浏览器会把它当搜索词交给默认搜索引擎。

---

## 2. DNS 解析：域名 → IP

浏览器需要知道 `example.com` 对应哪个 IP 地址，才能建立连接。查询顺序：

1. **浏览器 DNS 缓存** — 最近访问过的域名会缓存，直接命中就跳过后面
2. **系统 DNS 缓存**（`/etc/hosts` 文件 + 系统缓存）
3. **路由器缓存**
4. **ISP 的递归 DNS 服务器** — 向上级查询，最终找到权威 DNS 服务器返回结果

**可能出的问题：**
- DNS 解析慢：通常 20-100ms，跨地区可能更慢
- 解决方案：`<link rel="dns-prefetch" href="//cdn.example.com">` 提前解析第三方域名

---

## 3. 建立 TCP 连接（三次握手）

拿到 IP 后，浏览器和服务器建立 TCP 连接，需要三次握手：

```
客户端 → SYN           → 服务器   // 我要连接
客户端 ← SYN + ACK     ← 服务器   // 好的，我准备好了
客户端 → ACK           → 服务器   // 收到，开始传输
```

每次握手都有网络延迟，距离越远延迟越大。这就是为什么 CDN 能加速——让用户连接距离更近的节点。

**HTTPS 还要多一步 TLS 握手**，协商加密算法、验证证书，再额外增加 1-2 个 RTT（往返时间）。

**HTTP/2 的优化**：多路复用，一个连接可以并发多个请求，不需要像 HTTP/1.1 那样排队等待或建多条连接。

---

## 4. 发送 HTTP 请求

TCP 连接建立后，浏览器发送 HTTP 请求：

```
GET /index.html HTTP/1.1
Host: example.com
Accept: text/html
Cookie: session=xxx
Cache-Control: max-age=0
```

**缓存机制在这里起作用：**
- 强缓存（`Cache-Control: max-age`）：未过期直接用本地缓存，不发请求
- 协商缓存（`ETag` / `Last-Modified`）：发请求，服务器返回 304 时用本地缓存

这是前端能做的最简单高效的优化之一。

---

## 5. 服务器处理并返回响应

服务器收到请求后处理，返回响应：

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Encoding: gzip

<!DOCTYPE html>...
```

**可能出的问题：**
- TTFB（Time to First Byte）过长：服务器处理慢，或网络传输慢
- 响应体太大：没有开 gzip/brotli 压缩

DevTools → Network → 点击请求 → Timing 面板，可以看到每个阶段耗时，TTFB 对应 "Waiting (TTFB)" 那一行。

---

## 6. 浏览器解析 HTML

收到 HTML 后，浏览器开始边下载边解析，构建 **DOM 树**：

```
HTML → 词法分析（Tokenizer）→ 语法分析 → DOM 树
```

**解析遇到资源时：**

- `<link rel="stylesheet">` — 下载 CSS，**阻塞渲染**（不阻塞 HTML 解析）
- `<script>` — **阻塞 HTML 解析**，下载并执行完才继续（除非加了 `async` 或 `defer`）
- `<img>`、`<video>` — 不阻塞解析，异步下载

这就是为什么：
- CSS 放 `<head>`：尽早下载，避免 FOUC（无样式内容闪烁）
- JS 放 `<body>` 底部或加 `defer`：避免阻塞 HTML 解析

```html
<!-- defer：HTML 解析完再执行，保证顺序 -->
<script src="app.js" defer></script>

<!-- async：下载完立即执行，不保证顺序 -->
<script src="analytics.js" async></script>
```

---

## 7. 构建 CSSOM

下载完 CSS 后，浏览器解析 CSS 构建 **CSSOM 树**（CSS Object Model）。

CSSOM 构建完成前，浏览器不会开始渲染——这是 CSS 阻塞渲染的原因。CSS 文件越大、层级越深，CSSOM 构建越慢。

---

## 8. 合并为 Render Tree，布局，绘制

DOM + CSSOM → **Render Tree**（只包含可见节点，`display: none` 的不在里面）

然后：

1. **Layout（布局/重排）**：计算每个节点的位置和尺寸
2. **Paint（绘制/重绘）**：把节点转成像素
3. **Composite（合成）**：把多个图层合并，送给 GPU 显示

**重排（reflow）代价最高**，改变了元素的几何属性（宽高、位置）会触发。**重绘（repaint）**次之，只改变了外观（颜色、背景）但不影响布局。**合成**最便宜，`transform` 和 `opacity` 的动画只触发合成，不触发重排重绘，所以动画要用 `transform`。

---

## 9. JS 执行

JS 在主线程执行，会阻塞渲染。执行时间过长会导致页面卡顿。

**可能出的问题：**
- 长任务（Long Task）：单个 JS 任务超过 50ms，用户操作感觉卡
- 解决方案：把长任务拆成小任务（`setTimeout(fn, 0)` 或 `scheduler.postTask`）、用 Web Worker 把计算移到后台线程

---

## 用这个框架排查问题

| 现象 | 可能的环节 | 排查工具 |
|---|---|---|
| 页面很久才开始显示 | DNS / TCP / TTFB | DevTools Network → Timing |
| 首屏白屏 | JS 阻塞渲染、CSS 未加载 | DevTools Performance → FCP |
| 滚动/交互卡顿 | 主线程长任务、重排 | DevTools Performance → Long Tasks |
| 资源加载失败 | 网络、CORS、缓存 | DevTools Network → 看状态码和报错 |
| 重复请求 | 缓存未配置 | DevTools Network → 看 Size 列是否显示缓存 |

Chrome DevTools 的 **Performance** 面板录制一次完整加载，能看到每个阶段的时间占比，是定位问题最直接的方式。

---

## 平时写代码该注意的

理解了这个流程，很多「规范」背后的原因就清楚了，不是死记硬背，是自然而然的习惯。

**第三方资源加 dns-prefetch。** 页面里用了 CDN、埋点、字体这些第三方域名，提前告诉浏览器去解析 DNS，用户真正触发请求时就能省掉这段时间：

```html
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="preconnect" href="//api.example.com">
```

`preconnect` 更激进，连 TCP 握手都预做了，适合你确定一定会用到的关键域名。

**JS 加 defer，不要放 `<head>` 里裸写 `<script>`。** 没有 `defer`/`async` 的脚本会直接阻塞 HTML 解析，页面内容要等 JS 下载执行完才能继续渲染。现代框架的构建工具一般会处理这个，但手动加脚本时要注意。

**CSS 不要写得太深。** CSSOM 构建完才开始渲染，选择器嵌套越深匹配越慢。实际影响通常不大，但刻意避免 `.a .b .c .d .e` 这种七八层的写法是好习惯。

**动画用 `transform`，不用 `top`/`left`。** 改 `top`/`left` 触发重排，代价最高。`transform` 只走合成阶段，GPU 处理，丝滑且不影响其他元素。这一点影响非常直观，动画卡不卡在这里差距很大。

```css
/* ❌ 触发重排 */
.moving { left: 100px; }

/* ✅ 只触发合成 */
.moving { transform: translateX(100px); }
```

**避免强制同步布局。** JS 读取布局属性（`offsetWidth`、`getBoundingClientRect`）会强制浏览器立刻做一次布局计算。如果在循环里交替读写，会反复触发重排：

```javascript
// ❌ 每次循环都触发重排
for (const el of elements) {
  const width = el.offsetWidth   // 读，强制布局
  el.style.width = width + 'px' // 写，使布局失效
}

// ✅ 读写分离
const widths = elements.map(el => el.offsetWidth)  // 集中读
elements.forEach((el, i) => { el.style.width = widths[i] + 'px' })  // 集中写
```

**首屏不需要的 JS 不要在首屏加载。** 知道了 JS 阻塞渲染这个原理，路由懒加载、`dynamic import` 的意义就很明确了——首屏只加载首屏需要的，其他的等用到再下载。

**图片加 `loading="lazy"`。** 浏览器处理 `<img>` 时会发起请求，首屏以外的图片加上懒加载，减少首屏网络竞争：

```html
<img src="photo.jpg" loading="lazy" alt="">
```
