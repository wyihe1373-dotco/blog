## 技术选型

H5 与 Native 通信常见三种方案：

- **URL Scheme 拦截**：Native 截获 `myapp://xxx` 请求，单向、传参不方便、无法同步返回
- **消息注入（JavascriptInterface）**：Android WebView 可直接注入 Java 对象，iOS WKWebView 不支持
- **DSBridge**：开源库，封装了 Android 注入 + iOS prompt 两种底层，对外暴露统一 API，支持同步和异步调用

选 DSBridge V3 的核心原因是**支持同步调用**。某些场景（比如在 JS 里获取 Native 的 Token 再发请求）必须是同步的——用 Promise 链会产生大量状态管理代码。其次，它的双端实现已经稳定，避免自己维护两套底层适配。

---

## 为了解决什么问题

项目 H5 需要运行在四个宿主：

| 宿主 | 通信协议 |
|------|---------|
| iOS/Android 主客户端 | DSBridge V3 |
| 旧版客户端 | 旧版私有 bridge 协议 |
| 桌面端 | `window.postMessage` |
| PWA / 纯 Web | `window.postMessage` |

如果每个业务模块自己判断 `isNativeApp()` / `isDesktop()` 再选协议，代码里会充斥环境判断逻辑，每次新增宿主都要改遍全量业务代码。

目标是让业务侧只写这一行，不感知宿主差异：

```typescript
const res = await AppSDK.Bridge.call<UserInfo>('getUserInfo', { uid })
```

## 实际开发中碰到的问题

### 问题一：WKWebView 不支持同步 JS → Native 调用

iOS WKWebView 出于安全限制，禁止 JS 直接同步调用 Native 方法。只有 `window.prompt` / `window.confirm` 是同步阻塞的——DSBridge 利用这个特性：用 `prompt('_dsbridge=' + method, argStr)` 传递调用，Native 拦截 `prompt` 事件并在同一调用栈里返回结果。

但这也带来一个新问题：某些第三方 SDK（广告、统计）会覆盖 `window.prompt`，导致 bridge 调用静默失败。最终方案是在 SDK 初始化时检测 `window.prompt` 是否被劫持，备用降级到异步回调模式。

### 问题二：SDK 初始化前的 Bridge 调用

页面加载时，某些业务代码会在 SDK 的 `initBridge()` 跑完之前就调用 `Bridge.call()`。如果 `Bridge` 是 `null`，直接 crash；如果是 `undefined`，报错。

解法是让 `Bridge` 的初始值是一组**安全空操作**——调用不 crash，但会返回 `{ err: true }`，让业务侧可以正常走错误分支：

```typescript
const Bridge = {
  call: () => Promise.resolve({ err: true, message: 'bridge not initialized', data: null, code: -1, version: '', build: '' }),
  callSync: () => ({ err: true }),
  register: () => {},
  hasNativeMethod: () => false,
}
```

SDK 初始化完成后，`initBridge()` 把真实实现注入进来。

### 问题三：PostMessage 的 `hasNativeMethod` 没有标准实现

桌面端没有 Native 注入，根本无法通过 API 查询"这个方法存不存在"。直接返回 `true` 会导致所有未实现的方法调用都挂起等待响应（永远不会来）。

解法是维护一个白名单 `CALL_LIST`，声明桌面端支持的方法名。`hasNativeMethod` 基于白名单判断，未注册的方法立即 resolve 并标记 `err: true`，不阻塞业务流程：

```typescript
if (!dsbridge.hasNativeMethod(handlerName)) {
  return resolve({ ...defaultResponse, err: true, message: `${handlerName} not support` })
}
```

## 解决方案：启动时注入，运行时透明

```typescript
// sdk/app/src/module/bridge/index.ts

export const initBridge = () => {
  if (isNativeApp()) {
    nativeDs.dsbridge.init()
    Bridge.call = nativeDs.call as BridgeCallHandler
    Bridge.register = nativeDs.dsbridge.register
    Bridge.hasNativeMethod = nativeDs.dsbridge.hasNativeMethod
  } else if (isLegacyApp()) {
    legacyDs.dsbridge.init()
    Bridge.call = legacyDs.call as BridgeCallHandler
  } else if (isDesktop() || isPWA() || isWeb()) {
    Bridge.call = msgDs.call as BridgeCallHandler
    Bridge.callSync = msgDs.callSync as BridgeCallHandler
  }
}
```

DSBridge V3 内部的 iOS/Android 分支：

```typescript
call(method, args, cb) {
  const arg = { data: args ?? null }
  if (typeof cb === 'function') {
    const cbName = `dscb${dsID}${window.dscb++}`
    window[cbName] = cb
    arg['_dscbstub'] = cbName
  }
  const argStr = JSON.stringify(arg)

  if (window._dsbridge) {
    // Android：直接调用 Java 注入对象（同步）
    return JSON.parse(window._dsbridge.call(method, argStr) || '{}').data
  } else if (navigator.userAgent.indexOf('_dsbridge') !== -1) {
    // iOS：通过 prompt 传递，Native 在同一调用栈返回
    return JSON.parse(prompt('_dsbridge=' + method, argStr) || '{}').data
  }
}
```

所有实现的返回值统一为同一结构，业务侧只需判断 `res.err`：

```typescript
interface BridgeCallResponse<T> {
  err: boolean
  message: string
  data: T
  code: number
  version: string  // App 版本号，用于版本兼容判断
  build: string
}
```

新增宿主时，只需在 `initBridge` 里加一个分支，所有 H5 App 的业务代码**零改动**。
