H5 页面图片多，这件事在做性能优化的时候体会很深。一次 Lighthouse 跑分，图片相关的问题占了一大半——格式没优化、尺寸没裁剪、没懒加载。从那次之后我把图片处理作为每个项目的标配，逐渐摸出了一套在 H5 里能用的方案。

## 原生懒加载够用吗？

浏览器自带的懒加载很简单：

```html
<img src="xxx.jpg" loading="lazy" alt="..." />
```

在现代浏览器里能用，但在嵌入 App 的 WebView 里不一定。我遇到过部分安卓 WebView 版本不支持的情况，图片直接全部加载了。

所以 H5 里我通常不单独依赖原生懒加载，会配合 IntersectionObserver 做一层兼容：

```ts
const useLazyImage = (src: string) => {
  const ref = useRef<HTMLImageElement>(null)
  const [loaded, setLoaded] = useState(false)

  useEffect(() => {
    const el = ref.current
    if (!el) return

    // 支持 IntersectionObserver 就用，否则直接加载
    if (!('IntersectionObserver' in window)) {
      el.src = src
      return
    }

    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting) {
          el.src = src
          el.onload = () => setLoaded(true)
          observer.disconnect()
        }
      },
      { rootMargin: '100px' }  // 提前 100px 开始加载
    )

    observer.observe(el)
    return () => observer.disconnect()
  }, [src])

  return { ref, loaded }
}
```

`rootMargin: '100px'` 表示元素进入视口前 100px 就开始加载，避免滚动时图片来不及加载的白屏。

## WebP / AVIF：格式选择

WebP 比 JPEG 体积小 25-35%，AVIF 更极端能小 50% 以上，但 AVIF 在部分 WebView 里支持不好。

用 `<picture>` 做渐进兼容：

```html
<picture>
  <source srcset="image.avif" type="image/avif" />
  <source srcset="image.webp" type="image/webp" />
  <img src="image.jpg" alt="..." />
</picture>
```

浏览器从上往下找第一个支持的格式，都不支持就回退到 `<img>`。

实际项目里，CDN 通常支持自动转换格式，不需要准备多份文件，请求时带参数就行（见下面 CDN 裁剪部分）。

## CDN 参数裁剪

这是性能提升最大的一个手段。后台传来的原图可能是 2000px 宽的大图，但在手机上展示只有 100px，传输的体积完全浪费了。

大多数 CDN（阿里云 OSS、七牛云等）都支持 URL 参数裁剪：

```ts
const getImageUrl = (originUrl: string, width: number) => {
  // 根据设备像素比计算实际需要的宽度
  const dpr = Math.min(window.devicePixelRatio, 3)
  const targetWidth = Math.round(width * dpr)

  // 阿里云 OSS 格式示例
  return `${originUrl}?x-oss-process=image/resize,w_${targetWidth}/format,webp`
}

// 使用
const imgUrl = getImageUrl(fundLogo, 48)  // 48px 显示，自动按 DPR 放大并转 webp
```

这样一来，1x 屏请求 48px 的图，3x 屏请求 144px 的图，格式自动转 webp。省流量非常明显。

## 关键图片预加载

首屏的关键图片（比如 banner、头像）可以用 `<link rel="preload">` 提前请求，避免页面渲染完才开始加载：

```html
<link rel="preload" as="image" href="banner.webp" />
```

在 Next.js 里：

```tsx
import Head from 'next/head'

<Head>
  <link rel="preload" as="image" href={bannerUrl} />
</Head>
```

只对首屏的关键图片做预加载，不要滥用，否则反而占用带宽影响其他资源加载。

## 低质量占位图（LQIP）

把原图缩到很小（比如 10px 宽），base64 内联在 HTML 里，先展示模糊的占位，等高质量图加载完再替换：

```ts
// 内联一个极小的 base64 占位图（通常 < 1KB）
const LQIP = 'data:image/jpeg;base64,/9j/4AAQSkZJRgAB...'

const FundBanner = ({ src }: { src: string }) => {
  const [imgSrc, setImgSrc] = useState(LQIP)

  return (
    <img
      src={imgSrc}
      style={{ filter: imgSrc === LQIP ? 'blur(8px)' : 'none', transition: 'filter 0.3s' }}
      onLoad={(e) => {
        if (imgSrc === LQIP) {
          const img = new Image()
          img.src = src
          img.onload = () => setImgSrc(src)
        }
      }}
    />
  )
}
```

## 在 Next.js 静态导出里的注意事项

这个项目用的是 Next.js + `output: 'export'`（静态导出），`next/image` 的自动优化在这个模式下不工作，必须设置 `images: { unoptimized: true }`。

所以优化工作都要自己做：用 `<img>` 标签，手动拼 CDN 参数，手动处理懒加载。

```tsx
// ❌ 不能用 next/image 的自动优化
import Image from 'next/image'
<Image src={src} width={100} height={100} />

// ✅ 用原生 img + CDN 参数
<img
  src={getImageUrl(src, 100)}
  loading="lazy"
  width={100}
  height={100}
/>
```

虽然麻烦一点，但显式控制反而更清楚，不会有黑盒行为。
